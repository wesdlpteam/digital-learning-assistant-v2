# Planner-context "soup" + tech-picker off-topic suggestions (2026-06-04)

## Symptom
A teacher used the public **"Have a tech tool in mind"** picker, chose **Lego Spike Prime**
for **Glen Waverley Year 5 "How We Organise Ourselves"** (a money & economics unit), and got a
suggestion entirely about **natural disasters / design-thinking / emergency-response robots**,
marked **"STRONG FIT"**. The generated text even cited a central idea of *"people design for
purpose"* and a line of inquiry about *"ethical and practical considerations in design"* — neither
of which belongs to an economics unit.

## Root cause (verified)
- The picker (`doGet ?action=suggestTech` → `suggestTechForPlanner_` → `loadPlannerContextForUnit_`)
  fed the AI the unit's **`plannerContextRich`** field and *preferred it over* the unit's
  `ci`/`lo`.
- **`plannerContextRich` is unreliable corpus-wide.** It holds the *whole year-level* combined
  planner ("the soup" of every theme in that year), copied onto every unit in the year — it was
  written by the old whole-file `enrichPlannerContext`, and the section-scoped re-extraction that
  was meant to replace it (`extractUnitsFromCombinedPlanners`) **skips any unit that already has
  `ci`+`lo`** (Code.js ~2448), so the soup is frozen in place.
- Measured against `data.json` (134 units): the per-unit blob's own leading `Theme:` marker /
  WiSE header disagrees with the unit's theme for ~117 of 134 units — i.e. nearly the whole corpus
  carries year-level-soup or wrong-section text in `plannerContextRich`.
- **`ci`/`lo` are reliable** (133/134 populated and correct), which is why the *normal* suggestion
  list (the `s` array, generated by the "inspiring" pipeline off `ci`/`lo`/`plannerText`) is mostly
  on-topic. The picker was the one teacher-facing consumer of the bad blob.
- A secondary contributor: `_matchSectionToTheme_` did a loose first-match-wins `includes()` scan,
  so even a clean re-extraction could confuse **HWEO** (Express) with **HWOO** (Organise) — one
  transposed letter — and match the short acronym **WWA** inside **WWAIPAT**.

## Fixes shipped this commit (code only — no data changed)
1. **`loadPlannerContextForUnit_`** now builds the picker's context from the unit's **Central Idea
   + Lines of Inquiry** (authoritative), not `plannerContextRich`. Fixes off-topic suggestions for
   **every** unit with no data surgery. Falls back to the blob only when `ci`/`lo` are both empty.
2. **Honest fit.** The prompt now does a FIT CHECK FIRST: if the tool genuinely doesn't suit the
   unit's central idea/LOI, it returns `fit:"poor"` with a plain explanation instead of a contrived
   activity. `index.html` renders a poor fit as an honest note + a **"Choose another tool"** button
   (no Value-Add / try-this scaffolding, no pinned chip).
3. **Matcher hardened** (`_canonTheme_` + `_matchSectionToTheme_`): canonical-theme match, so
   HWEO/HWOO and WWA/WWAIPAT no longer collide. Unit-tested.
4. **Cache bump** `tech_sugg_v2_` → `v3_` so old "forced fit" / soup-based answers are discarded.

## What is NOT fixed yet (needs a live Studio run — do deliberately)
The stored **`plannerContextRich`** soup and any **`s` suggestions already contaminated by a wrong
`plannerText`** (confirmed: GW Y5 HWOO's list still contains "Rescue Rover Rally: Design a Disaster
Response Fleet"). These live in Drive and can only be repaired in Studio.

Remediation plan (not yet run — carries the documented `s`-wipe / App-Smash regen risk):
1. Deploy the hardened backend (`clasp push` + `clasp deploy --deploymentId <ID>`).
2. Add a **force-re-extract** mode to `extractUnitsFromCombinedPlanners` that, for units that
   already have `ci`/`lo`, overwrites only `plannerContextRich` from the correctly-matched section
   and **preserves existing `ci`/`lo` and `s`** (never wipes suggestions unless `ci`/`lo` truly
   change). Run it per campus/year (resumable).
3. Re-derive `plannerText` (audit) for the repaired units, then **Bulk Regen** their `s`.
4. Spot-check GW Y5 HWOO + a sample per campus/year.
